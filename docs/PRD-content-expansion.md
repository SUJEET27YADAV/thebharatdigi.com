# PRD: Content Expansion — The Bharat Digital

## 1. Problem & Context

**Current state:** The Bharat Digital (thebharatdigi.com) is a Next.js App Router site with solid technical foundation — SEO metadata, schema JSON-LD, sitemap, WhatsApp CTA, product store, portfolio, and services pages. However, the content depth is thin across most pages. The site operates as a generalist web dev/IT agency but lacks the layered service pages, industry vertical pages, and supporting content that competitors use to capture qualified organic traffic.

**Why this matters now:** Analysis of 9 top-ranking competitors (Brandset Media, WebFX, Coalition Technologies, Digital Silk, BairesDev, Lounge Lizard, Designli, UPQODE, WPWeb Infotech, INOXOFT) reveals a universal pattern: every successful agency has 15–50+ dedicated landing pages, 3–6 industry vertical pages, FAQ sections per page, and blog/resource content. The Bharat Digital has ~15 indexable pages — none with FAQ schema, none industry-specific, and no resource content. Without equivalent depth, the site loses organic visibility for high-intent searches.

**Target customer overlap:** All 9 competitors target overlapping SME/founder audiences. WPWeb Infotech (India-based) and Brandset Media (Delhi NCR) are direct geographic competitors.

```
    page.tsx                           + LocalBusiness schema + city content
  guides/
    page.tsx                           + guide listing/download
  cost-calculator/
    page.tsx                           + interactive tool
```

### 5.2 Schema Requirements for Each New Page

- **Service pages**: `Service` + `BreadcrumbList` + `Organization`
- **Industry pages**: `LocalBusiness` + `BreadcrumbList` + `Service`
- **Blog posts**: Auto-generated by Ghost CMS (Article + BreadcrumbList)
- **Cost calculator**: `WebApplication` + `BreadcrumbList`

### 5.3 Content Management

For Phase 1–2 (service + industry pages), use static TSX with hardcoded content — no CMS needed for initial launch. Each page file is self-contained with metadata, schema, and components.

For Phase 3 (blog), use **Ghost CMS**:
- Self-hosted on Ubuntu laptop with SQLite, exposed via Cloudflare Tunnel
- Next.js fetches posts at build time via Ghost Content API (`@tryghost/content-api`)
- Blog routes (`/blog/`, `/blog/[slug]/`) use `generateStaticParams` for static generation
- Ghost handles the editor, SEO fields, images, and Article schema

**Why Ghost over MDX:** Rich editor experience, built-in SEO fields, image CDN, newsletter support ready for future growth. The Cloudflare Tunnel is free and requires no static IP.

## 7. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|---|
| Content feels templated/thin | Medium | Use unique angles per page; avoid copying any single competitor |
| Maintenance burden of 35+ new pages | Low | Static TSX pages = zero runtime cost; update only when needed |
| SEO dilution from too many thin pages | Low | Each page has unique metadata, schema, and 500+ words of content |
| Blog content doesn't rank immediately | Medium | Target long-tail, low-competition keywords initially |
| Navigation becomes cluttered | Low | Group new pages under dropdowns in Navbar |
| Competitors already dominate target keywords | High | Start with long-tail + local modifiers (e.g., "Noida" + "Next.js") to avoid head-term competition |
| City pages flagged as spam by Google | Low | Ensure each city page has unique local content, not templated |
